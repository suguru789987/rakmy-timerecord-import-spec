# 判断記録 (Judgment Log)

このドキュメントは、PdM (筒井) が AI コーディングアシスタント (Claude Code) と並走しながら本リポ `rakmy-timerecord-import-spec` (ラクミー タイムレコード CSV 取込フロー) の仕様策定・実装を進めた際に、**「AI が出した初期案」と「PdM が rollback または修正して採用した最終案」の差分とその理由** を時系列で記録するものです。

## なぜこの記録を残すのか

AI 補助環境では output 量が増える反面、**「どこが PdM の判断で、どこが AI 由来か」が外部から見えにくくなる** 課題があります。本記録はその透明性を確保し:

- PdM 固有の付加価値 (ドメイン知識・運用判断・UX 直感) を可視化
- 同じ判断軸を別の案件でも再現できるよう参照可能化
- 採用しなかった A 案も残し、「なぜ B が良かったか」の根拠を後から検証可能に

を目的としています。

## 運用ルール

1. **仕様変更時**: Claude が初期案を出し、PdM が rollback / 修正した場合は本記録に追記
2. **commit message**: 「`判断記録: rollback判断: A (旧) → B (採用), 理由: ...`」のフォーマットを採用
3. **A 案でも動いたケース**: 「結果論ではなく判定基準の根拠」を残すために A 案も記録
4. **半年に 1 回**: 本記録をレビューし、本リポ固有の判断軸を末尾セクションに抽出

## 判断詳細 (深掘り対象の主要 7 件)

### J-001 — エラー対応ボタンの均一化

**背景**: 取込プレビュー画面でデータ不整合行が出た時、ユーザーがどう修正するかの UI 設計が必要だった。マスタ未登録・値形式不正・列マッピング誤りなど、エラー種別が複数存在。

**Claude 提案 (A 案)**: エラー種別をシステムが判別し、最も適切なアクションを「緑色プライマリボタン」として強調表示。他の選択肢は「他の対応 ▼」セカンダリに格下げ。

**PdM 採用 (B 案)**: **全エラー行に同じ 3 ボタン** (➕マスタに追加 / ✏️行編集 / ⊘スキップ) を **均一サイズ・均一色** で並列表示。データ項目によって挙動 (押下時のモーダルや対象マスタ) だけが切り替わる。

**採用理由**:
- **判別不可エラー** が必ず存在する (システムが種別を確定できない場合の漏れリスク)
- 「3 ボタンの意味を 1 度覚えれば全エラー行で同じ操作」というユーザー学習コストの最小化
- システムの判別ロジック精度に運用が依存しない堅牢性

**振り返り**: 一度 Claude の推奨方式を実装した後、PdM が「判別不可エラーで漏れる可能性がある」と気づき rollback を指示した経緯がある。AI が「自分で考えた構造に従いがち」になる傾向への対抗判断として、本セッションで最も重要な意思決定。

---

### J-003 — マッピング画面はコンテキスト依存 1 画面

**背景**: プレビュー画面で「マッピング由来のエラー」を発見した時、修正のためにマッピング画面へ遷移する設計が必要。

**Claude 提案 (A 案)**: 「通常のマッピング設定画面」と「エラー修正専用マッピング画面」を物理的に別画面に分離。それぞれ責務を絞った独立画面を作る。

**PdM 採用 (B 案)**: マッピング画面は 1 つだけ。エラー由来で訪問した時のみ、上部に赤バナー (エラー文脈) + 該当マッピング行を赤枠ハイライトする **コンテキスト依存 UI** で対応。

**採用理由**:
- マッピングデータ自体は両画面で同じものを操作するため、画面分離すると **データ更新の整合性負債** が発生
- One-shot の URL パラメータ (`?goto=mapping`) で状態を渡し、適用後に URL から自動削除する設計で「リロード時の意図しない再発火」も防げる
- 画面数を増やすより、状態で UI モードを切り替える方が保守コスト低

---

### J-006 — 「自動適用」を「アクティブ」radio に変更

**背景**: マッピングプロファイル一覧画面で「次回以降この設定を自動で使う」フラグを設ける必要があった。同じ (種別 × 連携元) の組み合わせに複数のプロファイルが保存される想定。

**Claude 提案 (A 案)**: 「自動適用」というラベルの checkbox 列。各行で ON/OFF を独立に切り替え可能。

**PdM 採用 (B 案)**: ラベルを「アクティブ」に変更し、入力タイプを **radio (HTML `<input type="radio" name="mapping-active-{type}-{src}">`)** に変更。`(種別 × 連携元)` のグループ内で必ず 1 つだけが ON 状態になる。

**採用理由**:
- A 案では複数の checkbox を同時に ON にできてしまい、「同じ (種別, 連携元) で複数 ON の場合、次回取込でどれが自動適用されるか」が **UI からは判別不能** で矛盾する
- radio + `name` 属性のグループ化で **ブラウザが排他制御を強制** してくれるため、JS の追加ロジック無しで矛盾を排除可能
- 「複数候補を保存して radio で切り替える」運用が明示的に成立する

---

### J-007/J-008 — 判定列 3 種集約 + 状況詳細列で全行表示

**背景**: 管理者プレビュー画面では「4 分岐の自動判定」(新規 / 既存登録 / 差し替え / 既管理者) + エラーの 5 種類のステータスが存在。

**Claude 提案 (A 案)**: 判定列に 5 種類のステータスバッジを表示。pv-summary も 5 種類のボックスで件数を内訳表示。

**PdM 採用 (B 案)**:
- 判定列は **3 種類に集約** (🆕 新規 / 👥 既存 / ❌ エラー)
- サブ状態 (既存登録 / 差し替え / 既管理者) は **新設の「状況詳細」列** にテキスト + 色で表示
- 状況詳細列はエラー行だけでなく **全行で内部状態を表示** (新規登録 / 既存従業員 / 招待差し替え / 既管理者 / メール形式不正など)

**採用理由**:
- 5 種類のバッジを同列に並べると **「視覚的詳細すぎ」** で、3 種類カテゴリの大まかな状態把握ができない
- 「3 + 詳細」の階層構造により、ざっと見たい時と詳細を知りたい時の両方をサポート
- 状況詳細を **全行に表示** することで、「正常行も含めて何が起きるか」をプレビュー段階で確認可能 (= 取込実行前のレビューがしやすい)

### J-010 / J-011 / J-012 — UI 階層の整理 (進捗バー、ボタン配置、リンク位置)

これら 3 件は **「アクションフロー (必須経路) と 設定 (任意操作) を分離する」** という同じ原則からの判断:

**J-011 進捗バー**:
- A 案: 5 段階 (連携元 → 対象・マッピング → アップロード → プレビュー → 完了) でマッピングを含める
- B 案: **4 段階に削減** (連携元 → アップロード → プレビュー → 完了)、マッピングは進捗バーから外す

**J-012 切替リンク位置**:
- A 案: 情報バンド内に「⚙️ マッピング設定」と「📄 テンプレ取込に切替」を隣接
- B 案: 「テンプレ取込に切替」は **ページタイトル右上に独立配置** (モード切替の役割)、「マッピング設定」は情報バンドに残す (現モード内の設定)

**J-010 完了画面ボタン配置**:
- A 案: 「📧 招待メール一括送信」と「← 設定画面に戻る」が同じ緑塗りで中央並び
- B 案: 招待メールは緑塗りプライマリのまま、**戻るボタンは右端の灰枠 ghost に分離**

**共通する採用理由**:
- 「必須でユーザーが必ず通る経路 (= アクションフロー)」と「任意・条件付きの操作 (= 設定)」は **視覚的・位置的に区別すべき**
- 同じ重みで並べると「どれを優先すべきか」のメンタルモデルが作れない
- 設定系の操作はヘッダー右上やゴースト化で **「あるけど主役じゃない」** ことを示せる

### J-014 — ファイル選択の自動遷移を廃止

**背景**: ファイル選択ボタン押下時の挙動設計。複数アップロード口がある場合、自動で次の口へ移動するか、明示的にユーザーが進めるか。

**Claude 提案 (A 案)**: クリック → ファイルが選択された扱い → **次のアップロード口に自動遷移** → 最後の口でクリック → **プレビュー画面に自動遷移**。

**PdM 採用 (B 案)**: クリック → アップロードログ追加のみ (例: 「✓ 一時保存中 管理者_export.csv (12:51:04)」)。**自動遷移は一切しない**。「次へ: プレビュー →」ボタンを明示的に押した時だけ遷移する。

**採用理由**:
- 自動遷移は **誤操作のリスク** が高い (誤った口にクリックしてプレビュー画面に飛んでしまうなど)
- ユーザーが「今何をしているか」を **意識的に確認するチャンス** を確保
- 一度に複数ファイルアップロードした時、各ファイルの状態 (一時保存中の名前と時刻) を表示することで、何がどこで保存されたかが明示できる

### J-019 — 派生取込仕様の継承議論で「非推奨」を堅持

**背景**: 旧仕様で「顧客のカスタムテンプレ CSV を取り込んだ際に、システムが内部でフィールドを推測・仕分けして複数マスタ (店舗 / 雇用区分 / 従業員) を自動生成する」という派生取込機能の計画があった。

**Claude 提案 (A 案 / 旧仕様継承)**: 派生取込を実装し、顧客のテンプレ自由度を最大化する。

**PdM 採用 (B 案)**: **派生取込は非推奨**。顧客が「どの種別の CSV か」を明示宣言してから取込む方式 (per-type explicit upload) に統一。

**採用理由**:
- 顧客のカスタムテンプレは **列名・欠損・形式が無保証**
- システムによる自動仕分けは **必ず誤判定が混入** → 部分的に不完全なマスタが密かに生成される
- 後段の集計・打刻記録での不整合・トラブル原因の特定困難
- **エラー時の責任境界が曖昧** になる (顧客がやったのかシステムがやったのかが分からない)

この判断は **ドメイン (飲食 SaaS の運用実態) を理解していないと出せない判断** で、技術的には実装可能だが運用上は地雷、という典型例。

## 本リポで一貫して優先した判断軸

本セッションで現れた 20 判断から抽出される、PdM の判断原則:

1. **判別不可ケースで UI が破綻しない設計を優先** (推奨ボタン rollback、3 種カテゴリ集約)
2. **責務境界の明確化** (CSV取込設定 ≠ 外部連携、マッピング ≠ アクションフロー、サブ状態は別列で)
3. **冗長を削る** (バナー圧縮、状況詳細統合、不要トグル削除)
4. **ブラウザネイティブ UI を排除** (alert → トースト/モーダルに統一)
5. **明示的なユーザーアクション** (自動遷移より「次へ」明示クリック)
6. **状態管理の排他制御** (checkbox → radio、ブラウザ強制を活用)
7. **AI の「自分の構造に従いがち」傾向を見抜くメタ判断** (推奨ボタン rollback など)
8. **ドメイン知識による無効仕様の検出** (派生取込・型整合性・運用責任の境界線)
